## Credential lifecycle

### Issuance

When an operator submits a credential, Certiva:

1. Generates a unique `verificationId` and `verificationCode`.
2. Produces a compact QR payload (JSON) encoding the verification reference.
3. Computes a SHA-256 registry hash of the credential record.
4. Generates a metadata URI pointing to structured credential metadata.
5. Builds a public verification URL accessible without authentication.
6. Stores QR code image and metadata in Cloudflare R2 (or local filesystem in dev).
7. Optionally anchors the credential to the blockchain (Polygon Amoy) via the background worker.

### Verification

Public verification requires no authentication:

1. The verifier presents a verification code, scans a QR code, or visits the verification URL.
2. The API resolves the credential by code, ID, or QR reference.
3. The registry status is checked: `VALID`, `REVOKED`, or `NOT_FOUND`.
4. A structured result is returned to the verifier.
5. A `VerificationLog` entry is written with the verifier's IP and user-agent.

### Revocation

Operators with the `OWNER`, `SUPER_ADMIN`, or `ADMIN` role can revoke credentials. Revocation reasons:

| Reason | Description |
|---|---|
| `DATA_CORRECTION` | Credential contained incorrect data |
| `ISSUED_IN_ERROR` | Credential should not have been issued |
| `FRAUD_SUSPECTED` | Suspected fraudulent use |
| `INSTITUTION_REQUEST` | Institution policy decision |
| `OTHER` | Catch-all |
| `LEGACY` | Historical import |

Revoked credentials remain in the registry and return `REVOKED` on verification rather than being deleted.

## Document integrity proof

Certiva supports a separate document proof workflow independent of credentials:

1. The operator uploads a document (PDF or other file).
2. Certiva computes a SHA-256 hash and stores it as a `SecureDocumentProof` record.
3. The source file is discarded — only the hash is retained.
4. Verifiers submit the same document for comparison.
5. Certiva hashes the submission and compares it to the stored record.
6. The result is `VALID` (hash match) or `INVALID` / `NOT_FOUND`.

This mechanism allows any holder of the original document to prove it has not been altered, without the issuer needing to store the file.

## Admin role hierarchy

| Role | Capabilities |
|---|---|
| `OWNER` | Full control including admin management, settings, revocation, and audit log access |
| `SUPER_ADMIN` | Credential issuance, revocation, admin management, settings, audit logs |
| `ADMIN` | Credential issuance, document proofs, verification logs, read-only settings |
| `AUDITOR` | Read-only access to credentials, documents, verification logs, audit logs, and settings |

### Ownership constraints

- The last active `OWNER` account cannot be deleted, disabled, or demoted.
- Only an `OWNER` can promote another admin to `OWNER`.
- Non-`OWNER` admins cannot modify or delete `OWNER` accounts.
- An admin cannot disable their own account.
- Admins with any historical activity record can only be disabled, not deleted.

## Batch issuance

Operators can issue many credentials in a single operation by uploading a CSV file. The batch is processed asynchronously by the worker:

1. Operator uploads CSV via `POST /api/credentials/bulk`.
2. A job is enqueued on the `issuance` BullMQ queue.
3. The worker processes credentials from the CSV, generating all assets for each.
4. Results are tracked in an `IssuanceBatch` record, accessible via `GET /api/batches/:id`.
5. Failed rows are retried via the `retry` queue.

## Blockchain anchoring

Blockchain anchoring is an optional secondary integrity layer:

- Credential issuance and revocation events are recorded on the `CredentialRegistry` smart contract (Polygon Amoy testnet).
- Only the credential hash is stored on-chain — no personal data, student IDs, PDFs, or private metadata.
- The `credentialAnchor` BullMQ queue handles anchoring with 3 concurrent workers.
- Anchoring is non-blocking: credentials are usable immediately after issuance regardless of blockchain status.
- Blockchain audit logs are separately accessible via `GET /api/audit/blockchain`.

## Audit trail

Every significant action in Certiva produces an `AuditLog` entry. The audit log is append-only — there are no update or delete endpoints.

Tracked actions include:

- `LOGIN_SUCCESS`, `LOGIN_FAILURE`, `LOGOUT`
- `ADMIN_CREATED`, `ADMIN_UPDATED`, `ADMIN_DELETED`, `ADMIN_DISABLED`
- `CREDENTIAL_ISSUED`, `CREDENTIAL_REVOKED`, `CREDENTIAL_DELETED`
- `DOCUMENT_PROOF_CREATED`, `DOCUMENT_PROOF_DELETED`
- `SETTINGS_UPDATED`
- `FORBIDDEN_ATTEMPT`

## Storage abstraction

Asset storage (QR code images, credential metadata) is handled by a `StorageService` with two drivers:

| Driver | Environment | Notes |
|---|---|---|
| `local` | Development only | Stores files on the local filesystem |
| `r2` | Staging and production | Cloudflare R2 (S3-compatible object storage) |

The `r2` driver is enforced by environment validation in staging and production.
